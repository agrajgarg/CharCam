1. Product Summary

ASCII Camera is a browser-based creative utility. The main joy is instant transformation: the user grants camera permission and immediately sees their face, room, pet, object, or scene represented as moving text. The project should feel lightweight, private, and toy-like, but still useful enough for profile pictures, short clips, social posts, stream overlays, and visual experiments.

Core Promise

Live camera to ASCII

Convert camera frames into text characters in real time using brightness, contrast, edge detection, or color mapping.

User Feeling

Instant delight

The first session should be understandable in under 10 seconds: allow camera, see ASCII, tweak style, capture something fun.

Personal Scope

Small but expandable

Ship an MVP that is technically achievable, then add modes like posters, GIFs, terminal themes, and local galleries.

2. Goals and Non-Goals

Goals

  • Ask for camera access with clear permission messaging.
  • Show live ASCII preview at a smooth, adjustable frame rate.
  • Let users customize density, contrast, character set, color, and layout.
  • Support still image export in PNG and plain text formats.
  • Keep processing local in the browser for privacy and simplicity.
  • Work well on desktop first, with a usable mobile layout.

Non-Goals for MVP

  • No account system, cloud storage, or social network features.
  • No server-side video processing.
  • No complicated video editor timeline in the first version.
  • No perfect cross-device camera tuning on day one.
  • No native mobile app until the web version proves fun.

3. Target Users

The first-run flow should be short, honest, and recoverable. Camera permission failures are common, so the product needs useful empty states instead of a blank screen.

03

04

Customize

User adjusts density, palette, characters, mirror, and FPS.

05

Capture

User exports PNG, text, copied frame, or recorded clip.

5. MVP Feature Set

Feature Requirement Why It Matters
Camera Permission Use navigator.mediaDevices.getUserMedia(). Show permission, denied, no-device, and unsupported-browser states. Camera access is the app's main entry point, and permission problems must feel handled.
Live ASCII Renderer Render video to hidden canvas, sample pixels, map brightness to characters, draw output into a monospace preview. This is the core transformation and the first magic moment.
Density Control Slider for columns or resolution. Lower density improves performance; higher density increases detail. Users can make it readable on different screens and devices.
Contrast and Invert Basic image controls: contrast, brightness threshold, invert light/dark. Different lighting conditions need quick correction.
Character Sets Presets such as Classic, Blocks, Minimal, Code, Emoji-lite, and Custom input. Character choice changes the personality of the output.
Color Modes Monochrome green, amber terminal, grayscale, source-color, duotone. Gives the app visual range without making the UI complicated.
Capture Still Download PNG and copy ASCII text to clipboard. Makes the app useful beyond live play.
Privacy Notice State that camera frames are processed locally and not uploaded in the MVP. Builds trust before asking for a sensitive permission.

7. Controls and Interaction Model

Primary Controls

  • Start / Stop camera: clear permission entry point and a privacy-friendly stop button.
  • Density: adjusts output resolution.
  • Character set: dropdown plus custom character input.
  • Palette: swatches for terminal green, amber, grayscale, color, duotone.
  • Mirror: front-camera selfie mode should mirror by default.
  • Export: PNG, TXT, copy frame, and later GIF/WebM.

Keyboard Shortcuts

  • Space capture still frame.
  • M toggle mirror.
  • I invert output.
  • [ and ] decrease or increase density.
  • 1 through 5 switch theme presets.
  • R start or stop recording once recording exists.

8. Technical Approach

Rendering Pipeline

  • Request camera stream using browser MediaDevices API.
  • Attach stream to a hidden <video> element.
  • Draw each frame to an offscreen <canvas> at reduced resolution.
  • Read pixel data and compute brightness per cell.
  • Map brightness to a character ramp like .:-=+*#%@.
  • Render text using a <pre>, canvas text, or WebGL later if needed.

Recommended Stack

  • MVP: Vite + React or plain HTML/CSS/JS.
  • State: local component state; URL params for shareable presets later.
  • Export: canvas PNG for images, Blob download for text.
  • Recording: canvas capture stream plus MediaRecorder for WebM.
  • Persistence: localStorage for last-used settings.
  • Deployment: static hosting through Vercel, Netlify, GitHub Pages, or Cloudflare Pages.

9. Privacy, Safety, and Permissions

The app should earn the camera permission before it asks for it. Say what is happening, keep processing local, and make stopping the camera obvious.

Local by default

No camera frames leave the device in the MVP. The app can work fully as a static web page.

Visible camera state

Show a live indicator and a stop camera control whenever the stream is active.

Failure states

Handle permission denied, camera busy, no camera found, insecure origin, and unsupported browser.

11. Roadmap

Phase 1

MVP: camera permission, live ASCII preview, density, contrast, invert, character presets, still PNG and TXT export.

Phase 2

Better styling: color palettes, custom themes, mirror mode, keyboard shortcuts, local settings, responsive mobile layout.

Phase 3

Creative exports: GIF/WebM loops, poster generator, profile picture cropper, image upload conversion.

Phase 4

Power modes: stream overlay route, audio-reactive effects, motion trails, sharable preset URLs, gallery.

12. MVP Acceptance Criteria

Must Pass

  • User can grant camera access and see live ASCII output.
  • User can stop the camera stream.
  • User can adjust density and visual style without reloading.
  • User can export a still frame as PNG.
  • User can copy or download ASCII text.
  • App shows a helpful message when camera access fails.

Nice to Pass

  • Settings persist between visits.
  • Preview maintains acceptable performance on mobile.
  • Theme presets can be shared through URL parameters.
  • Image upload mode exists as a no-camera fallback.
  • Basic accessibility labels exist for all controls.

13. Open Questions

Product Decisions

  • Should the first version feel more like a creative toy or a utility?
  • Is recording important enough for MVP, or should still export come first?
  • Should the app include image upload from day one for users without camera access?

Technical Decisions

  • Should text render in a <pre> for authenticity or canvas for export fidelity?
  • What default density balances detail and performance best?
  • How much mobile support is expected for the first public version?

14. Suggested First Build

For a personal project, the best first build is a tight version that feels alive quickly. A strong weekend target would be:

  • One-page app with camera permission request and privacy note.
  • Live ASCII preview using hidden video and canvas sampling.
  • Controls for density, character ramp, invert, and palette.
  • PNG export by drawing the ASCII output to canvas.
  • TXT export and clipboard copy.
  • Local settings persistence.

After that, recording and theme packs are the strongest candidates for making it feel shareable.
